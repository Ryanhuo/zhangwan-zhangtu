# 需求与设计对齐指南

本指南用于新建或明显更新原型、主题、项目文档等资源时的统一协作规则。目标是先收敛产品需求，再确认设计方案，并把关键决策沉淀为可追溯快照。

## 适用原则

- 对齐贯穿全过程：读取资料、生成规格、计划实施和验收时，只要发现会改变方向、范围、成本或验收标准的问题，都要回到对应阶段继续确认。
- 方向优先于实现：先确认做什么，再确认怎么表达，最后再进入规格、计划和实施。
- 能从项目中读到的信息先读取，不把仓库里可自行判断的问题直接抛给用户。
- 局部文案、样式、素材替换、明确 bug 修复等不改变产品范围、信息架构和视觉骨架的任务，可以跳过正式对齐；但仍需记录本轮采用的关键假设，并在发现方向性缺口时回到对齐流程。

## 核心流程

```text
读取上下文 -> 产品需求对齐 -> 产品需求确认 -> DESIGN.md 候选确认 -> 设计方案对齐 -> 设计决策确认 -> 规格/计划 -> 实施验证
```

## 何时触发

- 新建原型页面、主题或项目文档。
- 明显重构信息架构、核心交互、页面流程或视觉方向。
- 用户需求模糊，例如只说“生成一个健身 APP”。
- 存在多种合理功能范围、布局方式、交互路径、内容组织或视觉方案。
- 用户明确要求先看方向、先出方案、先写规格或先出计划。
- 资料、主题、现有原型或资源目录之间存在冲突，且会改变产出方向。

## 上下文读取

提问前先读取可获得的信息，不把项目里能找到的问题抛给用户。

### 读取优先级

- 用户当前消息、附件、截图、链接和已给出的约束。
- 当前目录最近的 `AGENTS.md`、`README.md` 和相关规则文档。
- 关联原型：目录结构、入口文件、必要样式和已有交互。
- 资源目录：只做目录级扫描，了解已有原型、主题、文档和资产。
- 相关文件：只读取会影响本次判断的文件，不批量展开无关文件。

### 当前仓库的优先观察点

- `src/pages/`：当前页面原型事实源，优先查看是否已有同类页面、参考布局或可复用结构。
- `src/pages/*/spec.md`：页面级产品事实源。
- `src/pages/*/zhangtu.requirements.ts`：已沉淀的需求锚点与评审模块。
- `src/styles/` 与 `src/design-system/`：现有样式 token、设计系统和视觉约束入口。
- 页面目录内 `assets/`、`canvas-assets/` 等素材目录：如果存在，仅做与本次任务相关的读取。

### 扩展目录约定

当项目后续引入下列目录时，按以下规则纳入对齐流程：

- `src/prototypes/`：检查是否已有同类原型、参考页或可复用页面结构。
- `src/themes/`：检查是否已有相近主题或设计系统。
- `src/resources/`：检查是否已有业务说明、字段、流程资料或素材。

## 产品需求对齐

产品需求阶段负责回答“做什么”，方向应以收敛为主。通常需要明确：

- 目标用户和核心任务。
- 本次范围、功能清单和明确不做什么。
- 页面或资源的核心内容、数据来源和素材来源。
- 关键状态、核心路径和必要交互。
- 用户最终如何判断结果可用。

### 提问原则

- 只问会影响范围、成本或验收的问题。
- 能从上下文推断的内容直接记为假设并继续。
- 如果缺失信息会导致不同功能范围、不同页面结构或不同验收标准，必须先确认再继续。

## 设计方案对齐

设计方案阶段负责回答“怎么表达”。进入视觉、主题或明显 UI 改版前，必须先确认 `DESIGN.md` 设计基底。

### 设计基底规则

- 本项目设计基底固定为“掌玩UI”。
- 对应 skill 文件为 `.agents/skills/project/zhangwanUI/SKILL.md`。
- 对应设计事实源为 `.agents/skills/project/zhangwanUI/DESIGN.md`。
- 不再从多个主题中整理候选，也不为单个任务另起一套视觉系统。
- `DESIGN.md` 确定后，颜色、字体、布局、动效、组件形态等零散需求都作为该基底上的调整处理。

### 基底校验规则

- 优先读取 `.agents/skills/project/zhangwanUI/DESIGN.md` 作为视觉与组件事实源。
- 需要实现或消费设计系统时，再读取 `.agents/skills/project/zhangwanUI/SKILL.md` 与 `src/design-system/zhangwan-ui/SKILL.md`。
- 若用户提出的新要求与掌玩UI基底冲突，先确认是否需要更新该事实源，再进入实现。

### 设计问题覆盖要求

新建页面、大面积 UI 改版或模糊需求，设计阶段至少覆盖 5 个设计问题或设计决策变量；局部微调、明确 bug 修复或用户已给出完整设计约束时不受此下限限制。

常见设计问题包括：

- 首屏目标：主操作、信息概览、品牌表达或引导转化。
- 信息层级：哪些内容必须突出，哪些可以折叠或延后。
- 布局模式：看板、列表详情、仪表盘、向导、沉浸页或多步骤流程。
- 交互路径：高频用户效率优先，还是首次用户理解优先。
- 数据呈现：真实数据、文档推断、临时示例数据或空状态优先。
- 视觉语气：严格继承 `DESIGN.md`，还是在其基础上加强某种风格倾向。

### 落地约束

- 优先复用所选 `DESIGN.md` 对应的 CSS Variables、tokens 和组件习惯。
- 缺少 token 时，尽量贴近所选基底的视觉语言。
- 若存在无法从基底直接落地的部分，需在方案说明或交付说明中写明假设。

## 可视化对齐

- 对齐内容保持简洁，不做过度扩写。
- 优先使用轻量文本图帮助用户确认关键结构。

### ASCII Wireframe

- 新建页面、大面积 UI 改版或布局不明确时，必须先用 ASCII Wireframe 对齐页面布局，再进入实现。
- ASCII Wireframe 用于表达区域、层级、主次关系和关键操作位置，不替代最终视觉设计。

### ASCII Diagram

- 流程、结构、关系和图表类内容，可按需使用 ASCII Diagram 辅助沟通。
- 只有在它能帮助用户更快判断方向时才使用。

## 提问规则

- 先读上下文，能推断就不问。
- 默认一次只问一个决策点；同一主题下强相关的 2-3 个参数可以合并为一问。
- 尽量给出 2-3 个互斥选项，并标出推荐答案。
- 推荐答案必须说明取舍，不只表达偏好。
- 不为了补齐形式追问已能从上下文判断的问题。
- 用户回答后继续收敛；如出现新分支，再追加提问。

## 确认与记录

进入实现前，必须让用户确认产品需求和设计方案。确认产物可以是中文规格文档、执行计划或方案摘要，但至少包含：

- 产品需求：目标用户、核心任务、范围、功能清单、内容/数据来源、验收重点。
- 设计方案：采用的 `DESIGN.md`、整体设计方向、关键设计决策、设计假设。
- 实施边界：本轮会做什么，不会做什么。

### 归档规则

- 确认过的方案或规格必须归档为当时决策快照；未确认内容不归档。
- 原型相关内容保存到 `src/prototypes/<prototype-name>/.spec/`。
- 文件名必须包含日期，建议格式为 `YYYY-MM-DD-<topic>.md`。
- 归档内容只记录已确认的需求问题、设计问题、用户选择和最终设计决策。
- 归档文件只表示当下决策，不要求跟随后续实现持续同步。

### 当前仓库的兼容处理

- 在 `src/prototypes/` 尚未引入前，页面级确认文档可先放在对应页面目录内，遵循 `spec.md` 为事实源的现有约定。
- 若后续新增 `src/prototypes/`，新流程按本指南迁移，旧页面无需强制回填历史快照。

## 交付说明最小要求

只要本轮触发过对齐流程，最终交付说明或确认产物中至少补充以下三类信息：

- 问题。
- 用户选择。
- 最终设计决策。

## 与原型实现衔接

- 需求与设计决策确认后，再进入原型目录内实现、验收和修复阶段。
- 若任务落在 `src/prototypes/<name>/`，后续开发流程、多页面组织和验收以 [docs/prototype-development-guide.md](/Users/ryan/Desktop/zhangwan-zhangtu/docs/prototype-development-guide.md) 为准。
