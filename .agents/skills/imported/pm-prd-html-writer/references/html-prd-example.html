第一章 概述

1.1 背景与目标

随着企业业务的快速发展，用户权限管理成为保障系统安全性和数据合规性的核心环节。当前系统存在以下痛点：

  1. 权限控制粒度粗放，无法满足精细化管控需求
  2. 角色管理混乱，缺乏统一的权限模型
  3. 审计日志不完善，无法追溯操作记录
  4. 多系统权限分散，用户体验差

核心目标：构建一套统一、灵活、可扩展的用户权限管理系统，实现细粒度权限控制、完善的审计追踪和良好的用户体验。

3.2 角色权限功能

3.2.1 功能概述

角色权限模块采用经典的 RBAC (Role-Based Access Control) 模型，实现了用户-角色-权限的三层映射关系。系统预置了多个常用角色，同时支持管理员自定义扩展。

核心特性包括：

  • 角色继承机制：子角色自动拥有父角色的所有权限
  • 权限粒度控制：支持菜单级、按钮级、数据级三级权限
  • 动态权限加载：用户登录时按需加载，避免冗余
  • 权限变更生效：支持即时生效或下次登录生效两种模式

第四章 非功能需求摘要

4.1 性能要求

  • 页面加载性能
    • 首屏加载时间 ≤ 2 秒（3G 网络环境）
    • 可交互时间（TTI）≤ 3.5 秒
    • 资源大小限制：单页 HTML + CSS + JS 总计 < 500KB（gzip 后）
  • 接口响应性能
    • API 平均响应时间 < 200ms（P50）
    • API P95 响应时间 < 500ms
    • 并发支持：单实例 ≥ 1000 QPS
  • 数据处理性能
    • 用户列表查询（万级数据）< 300ms
    • 权限校验接口 < 50ms（需走 Redis 缓存）
    • 批量导入 500 条用户数据 < 10 秒

4.2 安全需求

  1. 身份认证安全
    • 密码必须使用 bcrypt 加密存储（cost factor ≥ 12）
    • 连续登录失败 5 次后锁定账户 30 分钟
    • 支持双因素认证（2FA）可选开启
    • Session/JWT Token 有效期 ≤ 2 小时
  2. 数据传输安全
    • 全站强制 HTTPS（HSTS 头部）
    • 敏感数据传输需额外 AES-256 加密层
    • API 接口实施 Rate Limiting（每 IP 每分钟 ≤ 120 次）
  3. 访问控制安全
    • 所有写操作必须有 CSRF Token 校验
    • 权限变更操作需要二次确认（短信/邮箱验证码）
    • 敏感操作（如删除）需审批流支持
  4. 数据安全
    • 个人隐私信息（PII）需脱敏展示
    • 数据库备份采用 TDE（透明数据加密）
    • 日志中禁止记录明文密码和完整 Token

⚠️ 安全红线：任何涉及用户权限变更的操作都必须记录完整审计日志，且日志不可篡改、不可删除（保留期限 ≥ 180 天）。

-- 用户表 DDL 定义
CREATE TABLE sys_user (
    user_id VARCHAR(32) NOT NULL COMMENT '用户唯一标识',
    username VARCHAR(50) NOT NULL COMMENT '登录用户名',
    real_name VARCHAR(100) DEFAULT NULL COMMENT '真实姓名',
    email VARCHAR(255) NOT NULL COMMENT '电子邮箱',
    phone VARCHAR(20) DEFAULT NULL COMMENT '手机号码',
    password_hash VARCHAR(255) NOT NULL COMMENT '密码哈希值 (bcrypt)',
    department_id BIGINT DEFAULT 0 COMMENT '所属部门ID',
    status TINYINT DEFAULT 1 COMMENT '状态: 0-禁用 1-正常 2-锁定',
    last_login_at DATETIME DEFAULT NULL COMMENT '最后登录时间',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (user_id),
    UNIQUE KEY uk_username (username),
    UNIQUE KEY uk_email (email),
    KEY idx_department (department_id),
    KEY idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='系统用户表';

-- 创建索引优化查询性能
CREATE INDEX idx_user_last_login ON sys_user(last_login_at);

// 创建用户 API 调用示例
const createUser = async (userData) => {
  const response = await fetch('/api/v1/system/users', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${getAuthToken()}`,
      'Content-Type': 'application/json',
      'X-Request-ID': generateUUID()
    },
    body: JSON.stringify({
      username: userData.username,
      real_name: userData.realName,
      email: userData.email,
      phone: userData.phone || null,
      department_id: userData.departmentId,
      role_ids: userData.roleIds || []
    })
  });

  if (!response.ok) {
    const errorData = await response.json();
    throw new Error(errorData.message || '创建用户失败');
  }

  return await response.json();
};

// 使用示例
try {
  const newUser = await createUser({
    username: 'zhangsan',
    realName: '张三',
    email: 'zhangsan@company.com',
    phone: '13800138000',
    departmentId: 1001,
    roleIds: [3]  // 普通用户角色
  });
  
  console.log('✅ 用户创建成功:', newUser.user_id);
} catch (error) {
  console.error('❌ 创建失败:', error.message);
}
